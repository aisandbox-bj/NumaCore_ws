Lens — Delivery Engine

The advisor's toolkit.

Five panels, one shared file, one session at a time. Lens loads the client JSON at session start, hosts the work that follows, and saves at session end. Never distributed to clients. Always running in the advisor's hands.

The five panels

Each panel owns one slice of the work.

Pulse is the morning view. Vitals goes deep on a single unit. Horizon plans the long arc. Cadence builds the schedule. Deploy executes it. They share a single source of truth — the client file — and they never write into one another's data.

Pulse · the morning view

What surfaces here.

Pulse is the home panel. It reads the same client file every other panel reads, and surfaces the units that need attention now — not because the heatmap is red across the board, but because a specific rule has fired. Pulse is opinionated: it does not show "everything," it shows what should be looked at.

  • Fleet health summary — total units, units in workshop, units flagged for attention
  • Attention items — condition-monitoring alerts at "very high" band, components past benchmark, PCRs missing work orders
  • Upcoming events — the next thirty days of scheduled work, with confirmation status
Cadence · the scheduling engine

Where the rolling plan lives.

Cadence is the longest-running tool in the suite. The rolling two-year replacement plan that anchors every Stride engagement is built and maintained here. Date forecasting projects each component forward from current SMU and the unit's utilisation profile; the optimiser sequences projects against resource conflicts; auto-build clusters components due in the same window on the same machine into proposed packages.

  • Schedule — configurable Gantt timeline of every upcoming PCR across the fleet
  • Optimiser — sequencing view; resource loading; conflict detection
  • Auto-Build — cluster components due in the same window on the same machine into project packages
  • Multi-cycle budget projection by fleet, year, and site
Deploy · execution layer

From scheduled to done.

Deploy is where confirmed projects become tracked work. Each project carries a structured work package — pre-shutdown checklists, day-of action items, meeting notes, phase tracking through the lifecycle. The terminology layer (Project / Shutdown / Outage) is configurable per client; the underlying data structure is the same.

  • Per-project checklists and action items, persisted in the client file
  • Phase tracking from approval through planning, execution, and closeout
  • Meeting notes attached to the project, not stored in a parallel system
Vitals · component health detail

Drill from fleet to component.

Vitals is the deep view. Heatmap of component status across the fleet; hero-view drill-down per unit with components arranged around the machine; condition-monitoring overlay when oil-sample data is present. The view a planner uses to decide which units need attention this week.

Horizon · long-range planning

The 3–5 year arc.

Horizon holds the long arc: utilisation profiles per unit, SMU forecast forward, fleet hours distribution, and the capital expenditure forecast that finance teams need to budget component replacements three to five years out.

Phase 4 · forthcoming

Signal — cross-client benchmarking.

Signal is the long play. Every Stride engagement adds component-life data to a growing benchmark dataset, keyed by deterministic component IDs and a controlled vocabulary. At five clients, Signal will answer "your 793 engine is running 12% below benchmark for this operating profile" with the data behind the claim. Not a Lens panel today — a separate tool, deferred until the data depth justifies the build.
